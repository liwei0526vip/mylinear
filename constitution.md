# 项目开发宪法

本文件定义了本项目不可动摇的核心开发原则。所有AI Agent在进行技术规划和代码实现时，必须无条件遵循。

---

## 第一条：简单性原则 (Simplicity First)
**核心：** 对于 Go 项目，遵循Go语言的"少即是多"哲学。绝不进行不必要的抽象，绝不引入非必需的依赖。
- **1.1 (YAGNI):** 只实现需求中明确要求的功能。
- **1.2 (标准库优先):** 优先使用Go标准库。
- **1.3 (反过度工程):** 简单的函数和数据结构优于复杂的接口和继承体系。

---

## 第二条：测试先行铁律 (Test-First Imperative) - 不可协商
**核心：** 所有新功能或Bug修复，都必须从编写一个（或多个）失败的测试开始。
- **2.1 (TDD循环):** 严格遵循"Red-Green-Refactor"循环。
- **2.2 (表格驱动):** 单元测试必须优先采用表格驱动测试（Table-Driven Tests）的风格。
- **2.3 (拒绝Mocks):** 优先编写集成测试，使用真实的依赖。

---

## 第三条：明确性原则 (Clarity and Explicitness)
**核心：** 代码的首要目的是让人类易于理解。
- **3.1 (错误处理):** **不可协商**：所有错误都必须被显式处理。错误传递时必须使用`fmt.Errorf("...: %w", err)`进行包装。
- **3.2 (无全局变量):** 绝不允许使用全局变量来传递状态，所有依赖必须通过函数参数或结构体成员显式注入。

---

## 第四条：UI/UX 设计一致性原则 (UI/UX Consistency) - 不可协商
**核心：** 本项目目标是克隆 [Linear](https://linear.app) 应用，前端界面的视觉设计、交互模式和用户体验必须与 Linear 保持一致。

### 4.1 视觉设计规范
- **布局结构**：三栏布局（左 56px 导航 + 中间自适应 + 右 384px 详情面板）
- **颜色系统**：主色调 Indigo (#6366F1)，严格遵循 `docs/竞品分析/11-UI-UX设计规范.md` 定义的颜色令牌
- **状态图标**：使用 Linear 标志性的 SVG 状态图标（虚线圆/空心圆/半圆/加号圆/勾选/叉号），对应 5 种状态类型
- **间距系统**：4px 基数（4/8/12/16/20/24/32/40/48/64px）
- **字体**：系统字体栈（-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto 等）

### 4.2 交互模式规范
- **命令面板**：Cmd+K 全局快捷入口，支持模糊搜索和分组命令
- **键盘驱动**：Vim 风格 J/K 导航，单键操作（C 创建/E 编辑/A 分配/L 标签/P 优先级/S 状态）
- **Peek 预览**：选中 Issue 右侧弹出预览面板，不离开列表上下文
- **拖拽排序**：看板视图和列表视图支持拖拽，使用 `position FLOAT` 字段
- **动画时长**：Fast 150ms (hover) / Normal 200ms (展开) / Slow 300ms (弹窗)

### 4.3 参考资源
- **主要参考**：`docs/竞品分析/11-UI-UX设计规范.md`（设计令牌、组件规格、动画标准）
- **交互参考**：`docs/竞品分析/02-功能模块.md`（界面结构、操作流程）
- **实时对照**：开发过程中应对照 [Linear 官方应用](https://linear.app) 验证设计一致性

### 4.4 界面语言规范
- **界面语言**：所有用户可见的前端界面（菜单、按钮、标签、提示、错误信息等）必须使用**中文**
- **技术术语**：常见技术术语可保留英文（如 API、Webhook、PR、MR、Commit、Issue、PRD、Spec）
- **快捷键提示**：快捷键组合可使用英文符号（如 `Cmd+K`、`Ctrl+C`）
- **代码与注释**：代码变量名、函数名使用英文，代码注释使用中文

### 4.5 实施要求
- 所有前端组件开发前，必须先查阅 `docs/竞品分析/11-UI-UX设计规范.md`
- 新增 UI 组件时，必须确保视觉风格与 Linear 一致
- 交互行为有疑问时，以 Linear 官方应用的实际行为为准
- 禁止"创造性"的 UI 设计，严格遵循 Linear 的设计语言
