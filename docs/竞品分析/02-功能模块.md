# 功能模块分析

## 2.1 Issues（事务/工单）

Linear 的最基本工作单元。

### Issue 详情面板结构

**顶部操作栏**
- Issue 导航（序号显示，如 2/3）
- 快捷操作图标：复制链接、分享、更多操作

**中间主内容区**
- Issue 标题（可编辑）
- 描述内容（支持 Markdown）
- 子任务列表（TODO 清单）
  - 支持勾选完成
  - 可添加新的子任务（"Add sub-issues"）
- 附件区域
  - 支持拖拽上传
  - 显示附件图标和添加按钮
- Activity 活动流
  - 显示创建者、操作时间
  - "Unsubscribe" 取消订阅按钮
- 评论区
  - "Leave a comment..." 输入框
  - 支持附件和格式化

**右侧属性面板（Properties）**
- Status（状态）：下拉选择，显示所有可用状态
- Priority（优先级）：Urgent / High / Medium / Low / None（从高到低）
- Assignee（负责人）：显示邮箱
- Labels（标签）：可添加多个标签（如 Bug）
- Project（关联项目）：显示项目名称
- Milestone（里程碑）：关联到具体里程碑
- Cycle（迭代）：关联到当前 Cycle
- Estimate（工作量）：T-shirt sizing 或数字
- Due date（截止日期）：日期选择器

### 创建 Issue 流程

**界面结构**
- 模态弹窗（居中显示，半透明背景）
- 不离开当前页面，快速创建

**顶部标题栏**
- Issue ID 预览（自动生成，如 MYD）
- Templates 按钮：选择 Issue 模板
- 全屏/关闭按钮

**主要输入区域**
- Issue title（标题输入框）
- Add description...（描述输入框，支持 Markdown）

**快速建议（Quick suggestions）**
- 显示常用的里程碑、标签等
- 点击快速应用

**属性设置栏（底部工具栏）**
- Status（状态）：In Progress / Todo / Backlog 等
- Priority（优先级）：Urgent / High / Medium / Low
- Assignee（负责人）：分配给团队成员
- Project（项目）：关联项目
- Milestone（里程碑）：关联里程碑
- Labels（标签）：添加标签
- ...（更多选项）

**底部操作按钮**
- 附件上传图标
- Create more：创建后继续创建下一个
- Create issue：确认创建（蓝色高亮）

**交互特点**
- 所有关键属性在一个页面设置
- 支持模板快速创建
- 支持批量连续创建
- 智能建议减少输入成本

### Issue 功能列表

| 功能 | 说明 |
|------|------|
| 基本属性 | 标题、Markdown 描述、状态、优先级（Urgent/High/Medium/Low/None） |
| 负责人 | 指定 Assignee，显示邮箱 |
| 标签 | 多标签分类，工作区级 + 团队级 |
| 截止日期 | Due Date |
| 工作量预估 | T-shirt sizing 或数字 |
| 子任务 | 内嵌 TODO 清单，支持勾选完成 |
| 嵌套子任务 | 列表中支持嵌套层级显示子任务 |
| 父子关系 | Parent Issue + Sub-Issues，支持自动完成联动 |
| Issue 关系 | Blocked by / Blocking / Related / Duplicate |
| Cycle 归入 | 归入迭代周期 |
| Project 关联 | 关联到项目 |
| Milestone 关联 | 关联到里程碑 |
| 附件 | 文件上传、图片拖拽 |
| 评论 | @mention、Markdown |
| 活动流 | 显示所有操作历史和时间线 |
| 状态停留时间 | Time in Status，追踪 Issue 在各状态停留的时间 |
| 订阅 | 自动/手动订阅变更，可取消订阅 |
| 历史记录 | 字段变更审计 |

| 批量操作 | 多选批量修改 |
| Issue 模板 | 预填充模板 |
| Issue SLA | 基于规则自动应用时限（所有计划） |
| 邮件创建 | 通过邮件创建 Issue |
| 快捷操作 | 复制链接、分享 |

### 场景与解决的问题

| 场景 | 功能支撑 | 解决的痛点 |
|------|---------|-----------|
| 快速记录需求 | 模态弹窗创建 + Cmd+K 命令面板 | 不离开当前上下文，3 秒内完成创建 |
| 工作量评估 | T-shirt sizing / 数字估点 | 团队对齐迭代容量，避免过度承诺 |
| 任务拆解 | Sub-Issues 树形层级 + TODO 清单 | 大任务拆小，明确各子任务负责人 |
| 跨任务协调 | Issue Relations（Blocking/Related/Duplicate） | 可视化依赖关系，提前识别瓶颈 |
| 合规追踪 | SLA 规则 + 火焰图标紧急程度 | 自动应用时限，防止错过 SLA |
| 批量处理 | 多选 + 批量状态/分配变更 | Sprint Planning 时高效归类工单 |

### 使用技巧

- 用 `C` 快捷键快速创建 Issue，配合 `Create more` 模式连续批量创建
- 利用 Issue 模板标准化 Bug Report / Feature Request 的结构
- 通过 `Time in Status` 追踪瓶颈（在哪个状态停留过久）
- 开启重复任务（Recurring Issues）自动化例行工作

### 最佳实践

- **Issue 标题规范**：使用「动词 + 对象 + 上下文」格式，如 "Fix login timeout on mobile"
- **优先级使用**：Urgent 仅用于生产事故，High 用于当前迭代必须完成项，避免"一切皆高优"
- **标签体系**：工作区级标签用于全局分类（Bug/Feature/Improvement），团队级标签用于领域分类（Frontend/Backend/Infra）

## 2.2 Workflows（工作流）

| 功能 | 说明 |
|------|------|
| 默认状态 | Backlog → Todo → In Progress → In Review → Done → Cancelled |
| 状态图标 | 每个状态有独特的图标和颜色标识 |
| 完整状态列表 | Backlog（虚线圆）、Todo（空心圆）、In Progress（黄色半圆）、In Review（绿色加号圆）、Done（蓝色勾选）、Cancelled（灰色叉号）、Duplicate（灰色叉号） |
| 自定义状态 | 团队可自定义状态名称和流转 |
| Triage 状态 | 新 Issue 进入分诊 |
| 自动流转 | 通过 Git 集成自动变更状态 |

### 核心设计

- **5 种状态类型**：`backlog` → `unstarted` → `started` → `completed` → `cancelled`
- **转换约束**：如 `Done → Backlog` 禁止（需先 Reopen），防止状态混乱
- **Git 自动流转**：PR 创建 → In Review，PR 合并 → Done

### 推荐配置场景

| 场景 | 推荐状态流 |
|------|---------|
| 标准开发团队 | Backlog → Todo → In Progress → In Review → Done |
| 包含测试阶段 | 增加 Testing 状态（`started` 类型） |
| 运维团队 | Triage → Investigating → Mitigating → Resolved |

### 最佳实践

- 限制 `started` 类型状态不超过 3 个，避免流程过于复杂
- 所有自定义状态必须归入 5 大类型之一，保证统计口径一致
- 优先启用 Git 自动流转，减少手动状态更新的遗忘
- 状态转换历史（`issue_status_history` 表）是后续做 Cycle 报告、SLA 达标统计的数据基础——务必从一开始就记录

## 2.3 Projects（项目）

### 项目详情页面结构

**顶部标签页**
- Overview（概览）
- Updates（更新）
- Issues（工单列表）

**左侧主内容区**
- Latest update（最新更新记录）
- Description（项目描述，支持 Markdown）
- Milestones（里程碑列表）
  - 显示里程碑名称、截止日期、完成进度百分比
  - 支持添加新里程碑

**右侧属性面板（Properties）**
- Status（状态）：In Progress / Planned / Paused / Completed / Cancelled
- Priority（优先级）：Urgent / High / Medium / Low / None
- Lead（负责人）
- Members（成员列表）
- Dates（日期范围）：开始日期 → 结束日期
- Teams（关联团队）
- Labels（标签）

**进度可视化（Progress）**
- Scope（范围）：总工单数
- Started（已开始）：进行中的工单数 + 百分比
- Completed（已完成）：完成的工单数 + 百分比
- 进度趋势图：折线图显示时间线上的进度变化

### 项目功能列表

| 功能 | 说明 |
|------|------|
| 项目 CRUD | 创建/编辑/归档 |
| 项目描述 | 富文本协同编辑，支持 Markdown |
| 项目状态 | Planned / In Progress / Paused / Completed / Cancelled |
| 进度跟踪 | 自动统计完成度，可视化进度图表 |
| 项目负责人 | Lead + 成员列表 |
| 里程碑 | Milestones，拆分项目为多阶段，显示进度百分比 |
| 日期范围 | 开始日期 → 结束日期 |
| 优先级 | 项目级别的优先级设置 |
| 跨团队项目 | 多团队 Issues 归入同一 Project |
| 项目依赖 | 可视化项目间依赖关系，标识时间线冲突 |
| 项目模板 | 预定义 Issues 快速创建 |
| 项目文档 | 内嵌文档（PRD/Spec） |
| 项目更新 | Latest update 记录，Updates 标签页显示完整历史 |
| 里程碑转项目 | 可以将 Milestone 转换为 Project |

### 场景与技巧

- **进度可视化**：自动统计 Issue 完成百分比 + 趋势折线图，产品经理一目了然
- **里程碑拆解**：将大项目拆为 Milestone，每个 Milestone 有独立进度追踪
- **跨团队协作**：多团队的 Issue 归入同一 Project，打破团队信息孤岛
- **项目更新通报**（Project Updates）：定期发布进展，替代冗长的站会汇报

### 最佳实践

- 每个 Project 指定一个 Lead + 明确的 Target Date
- 项目描述中嵌入 PRD/Spec 文档链接
- 善用项目状态（Planned/In Progress/Paused/Completed）主动管理项目生命周期
- 里程碑可转换为独立 Project，适应需求规模变化

### 何时创建 Project：产品设计理念

**Project 的本质定位**：Project 是"**有明确交付物的工作集合**"，而非简单的 Issue 容器。Linear 的设计哲学强调 Project 应该是**面向结果的组织单元**。

#### ✅ 适合创建 Project 的场景

| 场景 | 判断标准 | 示例 |
|------|---------|------|
| **产品特性开发** | 需要多个 Issue 协同完成，有明确的用户价值交付 | "用户登录系统 2.0"、"移动端支付功能" |
| **跨团队协作** | 涉及 2+ 团队，需要统一进度追踪和沟通 | "新版官网上线"（设计+前端+后端+运维） |
| **有明确里程碑** | 可拆分为多个阶段，每个阶段有独立验收标准 | "数据迁移项目"（方案设计→测试环境验证→生产迁移） |
| **需要对外汇报** | 需要定期向管理层/客户汇报进度 | "Q1 性能优化计划"、"客户定制开发" |
| **有时间约束** | 有明确的 Deadline 或发布窗口 | "黑五促销活动技术准备"、"合规审计整改" |
| **技术债清理** | 聚焦某个技术领域的系统性改进 | "遗留代码重构"、"数据库性能优化" |

#### ❌ 不适合创建 Project 的场景

| 场景 | 原因 | 替代方案 |
|------|------|---------|
| **日常运维工作** | 持续性工作，无明确结束点 | 用 **Label**（如 `Ops`）+ **Cycle** 管理 |
| **零散 Bug 修复** | 缺乏共同目标，仅因"都是 Bug"而聚合 | 用 **Label**（`Bug`）+ **Priority** 过滤 |
| **单个 Issue 可完成** | 工作量小，无需拆解 | 直接创建 Issue，必要时用 Sub-Issues |
| **临时性任务** | 一次性工作，无复用价值 | 用 **Cycle** 归类本周/本月任务即可 |
| **纯粹的分类需求** | 仅为了"归类"而非"交付" | 用 **Label** 或 **Team** 分类 |
| **长期持续的主题** | 跨越多个季度，边界模糊 | 用 **Initiative** 管理战略方向 |

#### 🎯 核心判断原则

Linear 的 Project 设计遵循"**SMART 原则**"：

1. **Specific（具体的）**：有明确的交付范围，不是"提升用户体验"这种模糊目标
2. **Measurable（可衡量的）**：能通过 Issue 完成度量化进度（如 80% 完成）
3. **Achievable（可达成的）**：工作量适中，通常 2-12 周可完成
4. **Relevant（相关的）**：所有 Issue 都围绕同一个业务目标
5. **Time-bound（有时限的）**：有明确的开始和结束时间

#### 📊 规模参考

| Project 规模 | Issue 数量 | 时长 | 适用场景 |
|-------------|-----------|------|---------|
| 小型 | 5-15 个 | 1-2 周 | 单一功能模块开发 |
| 中型 | 15-50 个 | 4-8 周 | 产品特性开发、技术升级 |
| 大型 | 50-150 个 | 2-3 个月 | 跨团队大项目，建议拆分 Milestone |
| 超大型 | 150+ 个 | 3+ 个月 | **不推荐**，应拆分为多个 Project 或升级为 Initiative |

#### 💡 实战建议

**反模式警示**：
- ❌ 为每个 Epic 都创建 Project（过度使用）
- ❌ 创建"2024 年所有 Bug"这种永久性 Project（边界不清）
- ❌ Project 中混杂不相关的 Issue（缺乏内聚性）

**最佳实践**：
- ✅ Project 名称用"动词 + 名词"格式，如"重构用户认证模块"而非"用户认证"
- ✅ 每个 Project 在创建时就明确 Lead、Target Date 和成功标准
- ✅ 定期清理已完成的 Project（Archive），保持活跃 Project 列表简洁
- ✅ 当 Project 范围扩大时，及时拆分 Milestone 或升级为 Initiative

## 2.5 Cycles（迭代/Sprint）

### Cycles 管理界面

**左侧导航**
- Cycles 入口在团队名下，与 Issues、Projects、Views 平级
- 子菜单：Upcoming（即将到来的 Cycle）

**All cycles 视图**
- 竖向时间线 + 圆点展示 Cycle 排列
- 左侧显示日期刻度（如 Feb 16 → Mar 1 → Mar 16）
- 垂直连线表示 Cycle 之间的顺序关系

**Cycle 状态**
- Upcoming：即将开始
- Planned：已计划
- Active：进行中（有 Issue 时显示 Cycle Graph）
- Completed：已完成

**Cycle 属性显示**
- 状态标签（Upcoming / Planned / Active）
- Capacity：团队产能利用率百分比（如 0% of capacity）
- Scope：分配的 Issue 数量（如 0 scope）

**Cooldown（冷却期）**
- Cycle 之间可设置冷却间隔（如 "2 weeks cooldown"）
- 允许团队在迭代之间有缓冲时间
- 用于回顾、规划、技术债务清理等

### Cycle Graph（进度图表）

Cycle 开始后自动生成，每小时更新：
- 灰色线：总范围（Scope）
- 黄色线：已开始（Started）
- 蓝色实线：已完成（Completed）
- 蓝色虚线：目标进度（Target）

### Cycles 功能列表

| 功能 | 说明 |
|------|------|
| 周期配置 | 时长、起始日 |
| 自动创建 | 按计划滚动创建 |
| 自动移转 | 未完成转入下一 Cycle |
| Cooldown 冷却期 | Cycle 之间可设置间隔缓冲 |
| Capacity 产能 | 显示团队产能利用率 |
| Scope 范围 | 显示分配的 Issue 数量 |
| Cycle Graph | 自动生成进度图表（类似燃尽图） |
| 进度报告 | 完成 vs. 遗留统计 |
| Documents 支持 | Cycle 可关联 Documents 和 Links |

### 独特设计

- **Cooldown（冷却期）**：迭代之间设置缓冲期，用于回顾、规划、技术债清理
- **Cycle Graph**：自动生成进度图表（Scope/Started/Completed/Target 四条曲线）
- **自动移转**：未完成 Issue 自动转入下一 Cycle
- **Capacity 产能**：显示团队产能利用率

### 最佳实践

- 建议 2 周 Cycle + 2 天 Cooldown
- 使用 Capacity 指标控制团队不过度承诺
- Cycle 开始前利用 Backlog → Cycle 分配完成 Sprint Planning
- Cycle 支持关联 Documents 和 Links，可在 Cycle 中嵌入规划文档

## 2.9 Views & Filters（视图与过滤）

### 列表视图结构

**顶部工具栏**
- 视图切换器（Workspace/Team 选择）
- 快速过滤标签页：All issues / Active / Backlog
- Filter 按钮：添加自定义过滤条件
- Display 按钮：控制列显示、分组、排序

**分组显示**
- 按状态分组（可折叠）
- 显示每组的 Issue 数量
- 支持的分组维度：Status / Priority / Assignee / Label / Project

**Issue 列表项显示**
- 状态图标（左侧）
- Issue ID（如 MYD-1）
- Issue 标题
- 右侧元信息：
  - 标签（Label）
  - 关联项目（Project）
  - 负责人头像
  - 日期

**交互功能**
- 点击展开 Issue 详情面板
- 支持多选（批量操作）
- 拖拽排序
- 添加新 Issue（+ 按钮）

### Peek 预览

在列表视图中选中 Issue 后，右侧弹出预览面板，无需离开列表即可查看完整内容。

**预览面板内容**
- Issue ID + 标题

**设计特点**
- 右侧浮动面板，不遮挡列表
- 信息密度高：元数据 + 完整描述 + 子 Issue 一览

## 2.15 用户个人资料（Profile）

### Profile 页面结构

**入口**
- Settings → Profile（左侧导航第二项，位于 Preferences 下方）

**Profile 信息区**

| 字段 | 类型 | 说明 |
|------|------|------|
| Profile picture | 头像上传 | 用户头像，支持上传自定义图片 |
| Email | 文本（带编辑图标） | 用户邮箱地址，支持修改，用于登录和通知 |
| Full name | 文本输入 | 用户全名，显示在 Issue 负责人、评论等位置 |
| Username | 文本输入 | 用户名（一个词，如昵称或名字），用于 @mention |

**Workspace access（工作区访问）**

| 功能 | 说明 |
|------|------|
| Delete workspace | 计划永久删除工作区（红色 Delete 按钮），不可逆操作 |

### Profile 功能列表

| 功能 | 说明 |
|------|------|
| 头像管理 | 上传/更换个人头像 |
| 邮箱管理 | 查看/修改登录邮箱 |
| 全名设置 | 设置用户显示名称（Full name） |
| 用户名设置 | 设置用户名（Username），用于 @mention 引用 |
| 工作区删除 | 计划永久删除工作区（需确认，危险操作） |

### 设计要点

- **头像**：在 Issue 列表、评论、负责人选择器等多处展示，是用户身份的核心视觉标识
- **Full name vs Username**：
  - Full name：完整显示名称，用于正式展示（如 Issue 详情面板的负责人）
  - Username：简短标识，用于 @mention（如 `@liwel0526vip`），要求单词格式
- **邮箱**：同时作为登录凭证和通知接收地址，修改需验证
- **工作区删除**：高风险操作，放在 Profile 页面的 Workspace access 区域，与日常操作隔离

### 最佳实践

- 上传清晰的个人头像（建议 ≥ 128x128px），提高团队协作中的辨识度
- Username 使用简短易记的名称，方便团队成员 @mention
- Full name 使用真实姓名，便于团队识别
- 工作区删除操作应由 Workspace Owner / Admin 执行，建议在删除前导出数据备份
